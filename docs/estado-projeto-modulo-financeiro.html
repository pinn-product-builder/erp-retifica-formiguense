1. Contexto

O frontend é Vite + React + TypeScript, com dados em Supabase (Postgres, RLS, Auth). As regras de arquitetura do projeto exigem serviços para acesso ao banco, paginação no backend e telas responsivas. Em vários ambientes, o histórico de migrations do repositório não coincide com o registrado no projeto remoto do Supabase, o que impede supabase db push sem erros — daí o uso recorrente do SQL Editor com scripts manuais idempotentes.

2. Estágio anterior (baseline)

Situação consolidada antes de aplicar no banco remoto os scripts manuais e antes das últimas melhorias de código descritas abaixo.

Banco de dados (remoto)

  • Grupo econômico: tabelas organization_groups, organization_group_members e função org_ids_in_same_group ausentes ou não aplicadas no projeto Supabase em uso pelo app — o código já esperava esse schema.
  • RLS financeiro: políticas antigas podiam depender de escopo por organização atual; o modelo novo prevê is_org_member(org_id) para permitir consultas consolidadas com .in('org_id', …).
  • cash_flow.is_intercompany: coluna inexistente no remoto, gerando erro nas queries do fluxo de caixa e dos KPIs que filtram intercompany.
  • Caixa por usuário: migration volumosa (bank_accounts.owner_user_id, cash_register_sessions, cash_closings.bank_account_id, etc.) podia estar parcialmente aplicada ou rerodada — exemplo típico de erro: constraint cash_closings_org_date_bank_account_key já existente.

Aplicação (código já no repositório)

  • Visão consolidada por grupo econômico: hook useFinancialOrgScope, seletor de escopo nas telas financeiras, serviços com effectiveOrgIds — funcional no código, porém dependente do schema e RPC no banco.
  • Contas a receber: filtros existentes (status, datas, cliente por combobox, OS, orçamento); busca textual genérica, quando existente, podia estar aplicada após paginação (inconsistente com a regra de paginar no backend).
  • Fechamento de caixa: fluxo de caixa por operador, abertura com saldo, fechamento com divergência — já modelado no código; consolidado somando fechamentos por data.

3. Estágio atual

Situação após aplicar os SQLs manuais no Supabase (quando indicado) e após as alterações de código integradas ao repositório neste ciclo.

Banco de dados (alvo operacional)

  • Grupo econômico: schema criável via supabase/migrations/20260508100000_organization_groups.sql ou supabase/manual/apply_organization_groups.sql (script manual idempotente).
  • RLS financeiro alinhado: supabase/manual/apply_financial_rls_is_org_member.sql (complemento para cash_flow, contas, DRE, categorias, etc.).
  • Intercompany no fluxo: coluna cash_flow.is_intercompany via supabase/manual/apply_cash_flow_is_intercompany.sql.
  • Caixa por usuário: supabase/manual/apply_cash_register_per_user.sql (espelha a migration; trecho da UNIQUE em cash_closings ajustado para rerodagem segura).

Aplicação — comportamento esperado hoje

  • Grupo econômico: com RPC e tabelas no banco, o usuário vê o seletor Empresa / Todas as empresas nas telas financeiras quando há mais de uma organização acessível no mesmo grupo.
  • Contas a receber: filtros no backend — valor da parcela (igualdade com amount), texto combinável (nome, CPF/CNPJ do cliente e número da NF), mais status, datas e demais filtros; parser monetário BR em src/lib/parseMoneyBr.ts.
  • Fluxo de caixa: lançamento pode ser marcado como Intercompany; lista pode ocultar/exibir esses lançamentos; coluna persistida em cash_flow.
  • Fechamento de caixa: cada login usa conta de caixa própria (getOrCreateUserCashAccount); abertura com saldo inicial; fechamento com cálculo de divergência; Consolidado de caixa soma fechamentos do dia e passa a exibir também sessões de caixa ainda abertas por operador.
  • Testes automatizados: Vitest cobre utilitários (ex.: parseMoneyBr, orgScope, OrganizationGroupService com mocks).

4. Artefatos no repositório (referência rápida)

  • supabase/manual/ — scripts para SQL Editor (deploy quando db push não aplicável).
  • supabase/migrations/ — versão versionada das mesmas mudanças (CI / ambientes alinhados).
  • src/hooks/useFinancialOrgScope.ts — escopo financeiro multi-org.
  • src/services/financial/organizationGroupService.ts — RPC org_ids_in_same_group.
  • src/services/financial/accountsReceivableService.ts — filtros de lista no backend.
  • src/pages/FechamentoCaixaConsolidado.tsx — consolidado + sessões abertas.
  • .gitignore — supabase/.temp/ ignorado (estado local do CLI).

5. Deploy do schema (Supabase)

Para novos ambientes ou após clonar o repositório: aplicar, na ordem que fizer sentido para o seu banco, os scripts em supabase/manual/ conforme dependências (ex.: extensões e funções como update_updated_at_column, is_org_member já devem existir).
