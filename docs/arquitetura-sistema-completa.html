Resumo executivo

O que é o sistema

ERP multi-módulo para retífica de motores e operações correlatas: ordens de serviço, orçamentos, compras (cotações, pedidos, recebimentos), estoque (peças, lotes, alertas), financeiro completo, fiscal, cadastros, dashboards e administração de usuários e perfis.

Estilo arquitetural

Frontend: monólito SPA. Backend: não há API REST proprietária em servidor dedicado; o backend é a plataforma Supabase (Postgres + Auth + Storage + Edge Functions). Regras críticas residem em SQL (funções, triggers, RLS) e em serviços TypeScript no cliente.

Não há cluster de microserviços autônomos (Kubernetes com N serviços independentes). As Edge Functions são unidades deployadas à parte, mas compartilham projeto Supabase, segredos e banco.

Modelo C4 (adaptado)

Nível 1 — Contexto

Atores: usuários internos (operadores, compras, financeiro, gestão), eventualmente super-admin. Sistema troca dados com Supabase Cloud (hospedagem Postgres, Auth, Storage, Functions). Integrações auxiliares: e-mail transacional (fluxos de senha / hooks), possivelmente SFTP/banco em rotinas de integração contábil/bancária conforme configuração do cliente (código em serviços financeiros).

Camadas do frontend (clean architecture do projeto)

Regra do repositório: pages → components → hooks → services. Validação de entrada com Zod nos serviços/schemas. Listagens paginadas no backend (Supabase .range).

Roteamento e proteção

Definido em src/App.tsx com react-router-dom v6. Lazy loading de todas as páginas para code-splitting.

Tipos de rota

  • Públicas: / (landing), /auth, /reset-password.
  • ProfileProtectedRoute: exige autenticação + permissões de perfil por rota (useProfilePermissions).
  • AdminRoute: gestão de usuários e perfis.
  • ProtectedRoute: autenticação genérica (ex.: super-admin, 404).

Fluxos de dados (exemplos)

Autenticação e tenant

  1. Usuário autentica via Supabase Auth (email/senha ou fluxo configurado).
  2. JWT enviado em todas as requisições; RLS no Postgres usa claims / funções auxiliares (is_org_member, etc.).
  3. OrganizationProvider define org_id ativo; dados filtrados por organização e perfil.

Lançamento no fluxo de caixa

  1. Página FluxoCaixa chama FinancialConfigService.getOrCreateUserCashAccount para conta de caixa do operador.
  2. useFinancial → CashFlowService.create insere em cash_flow (validações + bloqueio se dia já fechado para a conta).
  3. Opcional: flag is_intercompany persistida na linha.

Aprovação de orçamento (fluxo pesado)

  1. Hook useDetailedBudgets invoca Edge Function process-budget-approval.
  2. Function (Deno + service role) orquestra atualização de orçamento/OS, reservas, necessidades de compra e alertas.

Relatório CSV

  1. useReports → supabase.functions.invoke('generate-report').
  2. Function monta datasets por reportCode e devolve CSV (ex.: vendas, produtividade).

Plataforma Supabase (detalhe)

ProdutoUso no projeto
Postgres Modelo relacional extenso (centenas de objetos nas migrations); integridade referencial; funções PL/pgSQL.
PostgREST API REST automática: .from('tabela'), filtros, joins via foreign tables.
RPC .rpc('nome', params) para lógica transacional e relatórios parametrizados no banco.
Auth Sessão no cliente; políticas RLS com auth.uid(); super-admin via funções dedicadas.
Storage Buckets para fotos de OS (order-photos), NF, XML fiscal, arquivos de diagnóstico, etc.
Edge Functions Ver seção dedicada; runtime Deno no Supabase.

Migrations em supabase/migrations são numerosas; em ambientes com histórico divergente no painel, usa-se também supabase/manual/*.sql para alinhar schema manualmente.

Storage (objetos)

Upload/download via supabase.storage.from('<bucket>'). Exemplos identificados no código:

  • order-photos — fotos ligadas a OS (useOrderPhotos.ts).
  • Buckets de NF / fiscal — conforme migrations (invoices_storage_bucket) e useFiscal.ts.
  • Anexos diversos — orçamentos, diagnósticos, AP (apInvoiceFileService, useOrderDocuments, etc.).

Políticas RLS de Storage são definidas nas migrations; o anon key só permite o que as policies liberarem.

Segurança, multi-tenant e permissões

  • RLS: predominante em tabelas sensíveis; evolução para políticas baseadas em membro da organização (is_org_member) no financeiro.
  • Tenant: coluna org_id nas entidades operacionais; troca de contexto no app.
  • Perfis: useProfilePermissions restringe rotas e ações por página/configuração de sistema.
  • Super-admin: funções RPC específicas; painel dedicado.
  • Grupo econômico: visão consolidada opcional; RPC org_ids_in_same_group + tabelas de grupo.

Inteligência artificial e automação

Pacotes de modelo generativo (LLM)

Não há dependências de OpenAI, Anthropic, Google Gemini, LangChain ou similares no package.json nem imports correspondentes no código de aplicação analisado. O ERP em produção não aciona modelos de linguagem para decisões de negócio nas telas documentadas aqui.

Automação “inteligente” sem LLM

  • suggest_suppliers_for_part: ranking de fornecedores implementado em SQL no Postgres (histórico, critérios de negócio).
  • KPIs e alertas: funções como calculate_kpi_trend, fluxos de workflow e gamificação — regras determinísticas.

Ferramentas de desenvolvimento

lovable-tagger (devDependency) relaciona-se ao fluxo de UI/Lovable no desenvolvimento, não a IA em runtime para usuários finais.

Qualidade, build e testes

  • Build: vite build — bundle otimizado, lazy routes.
  • Lint: ESLint 9 + TypeScript ESLint.
  • Testes: Vitest em serviços e utilitários (financeiro, parse monetário, escopo de org, etc.).
  • Tipagem: TypeScript estrito; tipos do banco em types.ts (regenerar quando schema mudar).

Glossário

RLSRow Level Security — filtros automáticos por linha no Postgres.
RPCRemote Procedure Call — função SQL exposta pela API Supabase.
PostgRESTCamada que transforma schema Postgres em REST.
Edge FunctionFunção serverless Deno rodando na infraestrutura Supabase.
TenantNeste projeto: organização (organizations) cujos dados são isolados por política.
